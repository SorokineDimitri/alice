Ce que fait le fichier

Tu lui donnes l'identifiant d'un livre Gutenberg (ex. Alice in Wonderland). Il renvoie deux listes triées par fréquence : les personnages (Alice, Queen, Hatter…) et les lieux (the garden, the kitchen, Wonderland…). Résultat mis en cache.

💡
Astuce centrale. spaCy sait déjà repérer des entités nommées (PERSON, GPE, LOC, FAC). Le rôle d'entities.py n'est pas de re-faire ce travail mais de filtrer le bruit et de rattraper les références déguisées comme « the Queen » ou « the garden ». Tout tient dans une seule règle par groupe, rangée dans un dictionnaire GROUPS.

Le pipeline en 5 étapes

Chaque entité repérée par spaCy passe par un mini-filtre, puis on compte.

  • 1
    run(book_id) — point d'entrée
    Regarde dans le cache. Si 11_entities.json existe → renvoie tel quel. Sinon lance le calcul.
    in 11 → out { characters, locations }
  • 2
    get_text(book_id) — récupérer le livre
    Lit le fichier local s'il existe, sinon télécharge depuis Gutenberg, puis nettoie l'en-tête/pied légal.
    in id → out texte propre (str)
  • 3
    nlp.pipe(spacy_chunks(text)) — laisser spaCy travailler
    Un livre dépasse souvent la limite spaCy. On le découpe en morceaux, et spaCy renvoie ses entités déjà repérées (doc.ents).
    in texte → out flux d'entités (PERSON, GPE…)
  • 4
    classify(entity) — ranger ou jeter
    Pour chaque entité, on parcourt GROUPS et on retient celui dont l'étiquette spaCy correspond ET (mot-clé connu OU contexte adapté). Sinon on jette.
    in entité → out "characters" / "locations" / None
  • 5
    Counter().most_common() — compter et trier
    Un Counter par groupe additionne les apparitions. most_common() trie automatiquement. Pas de tri manuel.
    in entités validées → out 2 listes triées

Deux groupes, une règle

Tout le filtrage tient dans classify() et le dictionnaire GROUPS. Pour ajouter un 3ᵉ groupe (objets ? animaux ?) → une entrée de plus, zéro changement de logique.

groupe 1

characters

quelqu'un qui agit dans le récit

Passe le filtre si :

  • label spaCy = PERSON, ET
  • contient un mot connu (queen, rabbit, hatter…) OU
  • un verbe (VERB/AUX) apparaît dans les 4 mots autour.
groupe 2

locations

un endroit, géographique ou physique

Passe le filtre si :

  • label spaCy = GPE, LOC ou FAC, ET
  • contient un mot connu (garden, kitchen, palace…) OU
  • une préposition (ADP : in, on, near…) apparaît autour.
⚠️
Pourquoi cette double règle ? spaCy se trompe dans les deux sens : il rate « the Queen » (souvent étiqueté NOUN) et il étiquette parfois mal des noms inconnus. Combiner label + contexte + mots-clés rattrape les deux types d'erreur sans complexité.

from __future__ import annotations
from collections import Counter

from modules.cache import load_json, save_json
from modules.nlp import load_spacy, spacy_chunks
from utils.path_config import cache_path, get_text

CONTEXT_WINDOW = 4
ACTION_POS = {"VERB", "AUX"}
LOCATION_POS = {"ADP"}

PERSON_NOUNS = { "baby", "cat", "queen", "hatter", … }
LOCATION_NOUNS = { "garden", "kitchen", "palace", … }

GROUPS = {
    "characters": {
        "labels": {"PERSON"},
        "nouns": PERSON_NOUNS,
        "context_pos": ACTION_POS,
    },
    "locations": {
        "labels": {"GPE", "LOC", "FAC"},
        "nouns": LOCATION_NOUNS,
        "context_pos": LOCATION_POS,
    },
}


def normalize(name):
    return " ".join(name.split()).strip(".,;:!?\"'“”‘’")


def context_tokens(entity):
    start = max(0, entity.start - CONTEXT_WINDOW)
    end = min(len(entity.doc), entity.end + CONTEXT_WINDOW)
    for token in entity.doc[start:end]:
        if token.i < entity.start or token.i >= entity.end:
            yield token


def has_context(entity, target_pos):
    return any(token.pos_ in target_pos for token in context_tokens(entity))


def contains_noun(entity, nouns):
    return any(token.lemma_.lower() in nouns for token in entity)


def classify(entity):
    for group, rule in GROUPS.items():
        if entity.label_ not in rule["labels"]:
            continue
        if contains_noun(entity, rule["nouns"]) or has_context(entity, rule["context_pos"]):
            return group
    return None


def find_entities(text):
    nlp = load_spacy()
    counters = {group: Counter() for group in GROUPS}

    for doc in nlp.pipe(spacy_chunks(text)):
        for entity in doc.ents:
            group = classify(entity)
            if group is None:
                continue
            name = normalize(entity.text)
            if name:
                counters[group][name] += 1

    return {
        group: [name for name, _ in counter.most_common()]
        for group, counter in counters.items()
    }


def run(book_id):
    path = cache_path(book_id, "entities")
    cached = load_json(path)
    if cached is not None:
        return cached

    result = find_entities(get_text(book_id))
    save_json(path, result)
    return result

✓
Le gros levier. Rassembler les règles dans GROUPS. À partir de là, has_context, contains_noun et classify deviennent les mêmes pour tous les groupes — on dit juste « voici une règle, applique-la ».

Et qu'est-ce qu'on perd ?

Toute factorisation est un compromis. Voici ce qui a disparu — et pourquoi c'est un bon échange.

context_bonus() — supprimé

Cette astuce poussait certains noms en haut du classement (+1 si le contexte matche). Mais les noms fréquents remontent déjà tout seuls — le bonus changeait surtout l'ordre des moins fréquents, qu'on ne montre pas dans la book card. Suppression sans regret.

possessive_locations() — supprimé

Cette fonction reconstruisait des phrases possessives à la main (« Queen's croquet-ground »). Joli, mais 40 lignes pour quelques entités en plus. spaCy capture déjà la plupart des lieux via GPE/LOC/FAC. Si un jour tu as besoin de très fine granularité, tu peux la rajouter dans GROUPS sous forme de règle séparée.

🎯
Règle d'or pour un junior. Quand tu vois deux fonctions presque identiques, demande-toi : « quelle est la différence ? » — souvent une seule valeur (un POS, une liste). C'est cette différence qui devient un paramètre, et la fonction devient unique.
